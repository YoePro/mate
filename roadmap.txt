MATE ROADMAP

Friend Mapping program

MATE is a personal relationship mapping system. The purpose is to help keep track of friends, family, organizations, shared history, activities, and relevant context. It should work like a small private CRM for friends and family, with a graph based interface inspired by Node RED but adapted for human relationships.

MATE can stand for:
Memories, Alliances, Time, Engagement

Core idea:

MATE is a cloud of connection: people are nodes, and their shared history, likely acquaintances, and relationship paths form a soft network around them. The goal is to capture the living network and its heritage, including those who are no longer with us.

* Remember people
* Remember how people are connected
* Remember where people belong
* Remember shared experiences
* Remember achievements, awards, roles, and career status
* Make it easier to follow up with people over time

Primary design principles:

* Local first
* Private by default
* Login required for normal use once account support exists
* Role based access from the start of account support
* Simple Go backend
* Vanilla HTML, CSS, and JavaScript frontend
* Neo4j as graph storage
* Minimal external dependencies
* Model personal metadata such as titles, awards, board memberships, competitions, and deceased status from the start
* main.go should stay very small
* Business logic belongs in internal/services
* HTTP logic belongs in internal/handlers
* Storage logic belongs in internal/storage
* Configuration belongs in internal/config
* Documentation should be kept close to the implementation

Version logic:

* 0.x odd versions are development versions.
* 0.x even versions are debug, test, and stabilization versions for the previous development version.
* 0.9 is a development version, not a beta gate.
* 0.10 is the debug and stabilization version for 0.9.
* 0.11 continues development, and later 0.x versions may continue this pattern until MATE is ready for 1.0.
* 1.0 = first stable release.
* After 1.0, version semantics can be revisited, but the same development/stabilization rhythm should remain available.
* Patch versions such as 0.1.1, 0.1.2, 0.2.1 are allowed for small corrections, documentation updates, cleanup, and minor internal adjustments.

Each version contains:

This file should describe what must exist, when it is considered done, and how it should be tested.

Access and account model:

MATE should support login before the first stable release. The practical model should stay simple:

* The first local account is created through a controlled bootstrap flow.
* The first account receives the owner role.
* The owner can create and manage other accounts.
* Roles should be explicit and small:

  * owner: full access, account management, configuration level operations.
  * admin: manage MATE data and invite or manage non-owner accounts if allowed by owner.
  * editor: create, update, and archive relationship data.
  * viewer: read-only access.
* Sessions should use secure HTTP-only cookies.
* Passwords must never be stored in plain text.
* Authentication and authorization must be enforced in services or middleware, not scattered through handlers.
* Single-user local use should remain easy, but not rely on an unauthenticated application once accounts exist.

Profile and attribute model:

MATE should treat people and organizations as rich profiles, not only graph nodes.

* A Person profile should support stable core fields such as name, nickname, gender, date fields where useful, deceased or legacy status, description, notes, and tags.
* A Person profile should support time-bound attributes such as titles, roles, education, certifications, awards, memberships, board roles, employment, competitions, and achievements.
* An Organization profile should support name, type, description, notes, tags, locations, contact or web references, aliases, active status, and time-bound roles or memberships.
* Time-bound attributes should support start date, end date, current status, source or notes, and optional relationship to an organization.
* Profiles should show direct relationships, related organizations, timeline-like attribute history, and graph context.
* The exact data model can evolve, but the API should not force everything into a single notes field.

Entity taxonomy model:

MATE should avoid treating every concrete thing as a top-level node type. Top-level graph entities should stay broad, with subtypes or classifications underneath where appropriate.

Candidate top-level entities:

* Person
* Organization
* Project
* Event, if events become graph entities rather than profile/context objects

Open modeling questions:

* Whether Location should be a top-level entity, a profile/context object, or both depending on use case.
* Whether Tag should be a graph entity or metadata/classification.
* Whether Family should be modeled as a group, organization-like entity, network, or relationship cluster.
* Whether Event should exist as a graph node or as an object attached to people, organizations, projects, or relationships.

Person subtypes and attributes:

* Person should remain the top-level entity.
* Gender should be modeled as a person attribute or field, not as separate top-level node types.
* Current known gender values are male, female, and other/unknown.

Organization subtypes:

Organization should remain the top-level entity, with subtype values such as:

* Company
* Association
* School
* Government
* Political Party
* Religious Organization
* Sports Club
* Military Unit
* NGO
* Community

Project entity:

Project should be considered its own top-level entity because it can be worked on, sponsored, owned, joined, paused, archived, and related to both people and organizations. Examples include:

* MATE
* GoFood
* Minecraft Server

Event modeling candidates:

Events may be useful as graph nodes or as attached objects. This should be decided by workflow need. Candidate event types include:

* Wedding
* Conference
* Birthday
* Meeting
* Trip
* Dinner
* School Reunion

Relationship attributes:

Relationships should be able to carry structured attributes where the relationship itself has important context. Example: `Person -> works_at -> Organization` may need `role: "Sales Manager"`, `start_date`, `end_date`, `current`, or notes. Common role examples include Sales Manager, Developer, Chairman, and Teacher.

Network and ownership model:

MATE should support user-owned networks before 1.0. A network is a user's bounded relationship map containing people, organizations, relationships, notes, context, and layout for a specific purpose.

* Each account can create one or more networks.
* A network has one owner account.
* Only the network owner can edit the network unless explicit sharing permissions are added later.
* Non-owners must not be allowed to edit another user's network data.
* Users should only see their own networks by default.
* Sharing must be explicit and permissioned, never implicit.
* Global person identity should be separated from network-specific notes, context, relationship meaning, and layout.
* The same global person may appear in multiple networks.
* Person creation inside a network should search for possible existing global persons before creating a new global person.
* Duplicate detection should produce suggestions with confidence score and matching reason.
* Matching can use name, nickname, organization, school, location, and known relationships.

0.9 Development

Goal:
Add user-owned networks and duplicate-aware person creation.

Scope:
This version introduces networks as first-class user-owned graph contexts. A user can create one or more networks, add existing or new people to a network, and keep network-specific notes and context separate from global person identity. Person creation should detect possible duplicates and suggest matches before creating a new global person. When duplicates are confirmed, MATE should support merging entities while preserving network ownership rules and moving non-conflicting properties to the surviving entity.
